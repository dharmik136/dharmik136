# Living Dossier README — Design Spec

**Date:** 2026-06-17
**Repo:** `dharmik136/dharmik136` (GitHub profile README)
**Status:** Approved design, pre-implementation
**Author:** Dharmik Shingala (with Claude Code)

---

## 1. Goal

Evolve the profile README from a static "intelligence dossier" into a **living case file**: every dynamic
element is an act of file-keeping (re-filing, re-stamping, status reconciliation), rendered entirely in the
existing monospace dossier voice. Nothing embeds a third-party widget/card — every visual is generated by our
own renderer, so the result reads as bespoke craft, not a copy-pasted template.

The footer line **"THIS DOSSIER IS LIVE"** becomes literally true: a GitHub Action ("the clerk") re-files the
dossier on a cadence.

**Non-goals (explicitly cut):**
- No external/self-hosted SVG endpoint (no Cloudflare Worker / Vercel). Committed assets + one Action only.
- No per-second/per-viewer real-time. "Live" means "re-filed at cron cadence" and the copy stays honest about that.
- No interactive "query desk" (Issue→Action loop) in this iteration.
- No contribution-graph gimmicks (snake, etc.).

## 2. Decisions (locked)

| Decision | Choice |
|---|---|
| Scope | Tier-1 only: committed SVGs + one GitHub Action, $0, no server |
| Build approach | **A — "The Clerk":** one Python renderer generates all SVGs + rewrites markdown sections |
| Motion | **Quiet.** §01 DOCTRINE types in once (~5s) then rests with a faint blinking cursor. Everything else static. |
| Status source | Track **private** repos too via a read-only fine-grained PAT (`DOSSIER_READ_TOKEN`) |
| Sections | §00 Integrity Seal, §00 Classification Header, §01 Doctrine typer, §02 Reconciled Status table, §07 Field Log |

## 3. Architecture — "The Clerk"

```
dharmik136/dharmik136
├─ README.md                  # restructured; embeds committed SVGs + marker sections
├─ dossier.yaml               # data: REF→repo map, doctrine lines, channels, classification words
├─ clerk/                     # Python renderer ("the clerk")
│   ├─ __init__.py
│   ├─ main.py                # entrypoint: gather → render → rewrite → (commit done by workflow)
│   ├─ data.py                # signals: git stats, GitHub API (REST/GraphQL), public events
│   ├─ svgtext.py             # text→vector-path outlining via fonttools (shared by all renderers)
│   ├─ render_seal.py         # → assets/seal.svg
│   ├─ render_doctrine.py     # → assets/doctrine-{light,dark}.svg   (SMIL typewriter)
│   ├─ render_header.py       # → assets/header-{light,dark}.svg     (UNCLASSIFIED / REDACTED)
│   ├─ render_status.py       # → assets/status-<REF>.svg            (per-project chits)
│   ├─ render_fieldlog.py     # rewrites the §07 markdown section text
│   └─ markers.py             # generic <!--SECTION:START/END--> replace-in-file
├─ assets/                    # committed generated SVGs (+ existing masthead-{light,dark}.svg)
└─ .github/workflows/dossier.yml
```

**Data flow:** `cron (daily) / workflow_dispatch` → `clerk/main.py` reads `dossier.yaml` + local git + GitHub API
→ writes SVGs into `assets/` and rewrites README marker sections → workflow commits **only if something changed**.

Each module has one job and a clear interface:
- `data.py` exposes `gather() -> DossierData` (a dataclass of all signals). Pure read; no file writes.
- each `render_*.py` exposes `render(data, cfg) -> None` writing its own asset(s); no cross-module coupling.
- `markers.py` exposes `replace_section(path, name, new_text)`; the only module that mutates README.md.

## 4. Components

### 4.1 `dossier.yaml` (the data contract)
Holds everything human-authored so the renderer stays generic:
- `doctrine`: the three rule strings.
- `refs`: list of `{ ref: DRS-001, title, domain, repo: owner/name, manual_status?: "..." }`.
- `channels`: LinkedIn, email.
- `classification`: light banner text, dark redaction words.
- `clerk`: cron cadence note, dormant threshold (default 30 days), field-log length (default 5).

### 4.2 §00 Integrity Seal — `render_seal.py` → `assets/seal.svg`
Wax-seal block: `ENTRY № 1 · REV <n> · SEAL <hex4> · FILED <YYYY·MM·DD>`. Static SVG, text outlined to paths.
- `REV` = content commit count (see §6 hash/rev rules).
- `SEAL` = first 4 hex of sha256 over README **with dynamic regions stripped** (§6).
- `FILED` = last content-commit date.

### 4.3 §00 Classification Header — `render_header.py` → `assets/header-{light,dark}.svg`
Two committed SVGs via `<picture>`. Light = `UNCLASSIFIED // FIELD FILE — SHINGALA, D.`. Dark = same geometry
with black `<rect>` redaction bars over a couple of words + faint `CONFIDENTIAL` stencil. **All text outlined to
paths** so redaction bars register exactly over words and fonts can't fall back.

### 4.4 §01 Doctrine typer — `render_doctrine.py` → `assets/doctrine-{light,dark}.svg`
The three doctrine rules type themselves in fixed-width, line after line, then hold. SMIL only:
- per-line reveal = `<animate>` on a clip-rect width; cursor = `<animate>` on opacity.
- `fill="freeze"`, **no `repeatCount` loop** on the reveal → text is permanent after ~5s (always readable).
- Glyphs rendered as outlined paths (stable clip widths, no font dependency).
- Light/dark pair via `<picture>`.

### 4.5 §02 Reconciled Status table — `render_status.py` → `assets/status-<REF>.svg`
Existing OPEN DOSSIERS table keeps REF/TITLE/DOMAIN/STATUS columns; the STATUS cell becomes an `<img>` chit
generated from real repo signals: `SHIPPED v0.1.0 · last activity 3d` (latest release tag, days since push).
- If no push in `dormant_threshold` days → faint diagonal `DORMANT` overprint (static).
- Private REFs read via `DOSSIER_READ_TOKEN`; public REFs work with `GITHUB_TOKEN`.
- If a ref has `manual_status` in yaml, that text is used verbatim (escape hatch for non-repo items, e.g. DRS-003).
- **No README marker needed:** the table cells are static `<img src="assets/status-<REF>.svg">` references written
  once; the clerk only regenerates the SVG *files*, so the README markdown for the table never changes.

### 4.6 §07 Field Log — `render_fieldlog.py` (rewrites README markdown, no image)
A tail of the last N (default 5) dated entries in our own dossier phrasing:
`2026·06·16 — DRS-002 · 3 commits · subject: APM boundary copy`, newest first, from public events.
- **Decay without color** (GitHub strips color/style in markdown): older lines shrink via `<sub>` and/or gain
  progressive `█` redaction. Never gray text.
- Dates come from event data, not run time (so delayed/skipped cron runs don't misdate entries).
- Public events only (private commits never appear here, by GitHub API design — acceptable).

### 4.7 Workflow — `.github/workflows/dossier.yml`
- Triggers: `schedule: "17 6 * * *"` (off top-of-hour congestion), `workflow_dispatch`.
- `permissions: { contents: write }`.
- Steps: checkout (full history for rev count) → setup Python → `pip install` (fonttools, PyYAML, requests) →
  run `python -m clerk.main` → commit **only if diff** → push.
- Commit author = `dossier-clerk[bot]`; message includes `[skip ci]`.
- Secret: `DOSSIER_READ_TOKEN` (fine-grained PAT, read-only metadata/contents/issues on the private REF repos).

## 5. README structure (target)

```
<picture> header-dark / header-light </picture>      ← §00 Classification Header
<picture> masthead-dark / masthead-light </picture>  ← existing masthead (kept)
<img seal.svg>                                        ← §00 Integrity Seal (image; not README text)
<picture> doctrine-dark / doctrine-light </picture>   ← §01 Doctrine typer
§02 OPEN DOSSIERS table … STATUS cell = <img status-DRS-00x.svg>  (static <img> refs; SVG files regenerated)
<!--FIELDLOG:START--> … §07 entries … <!--FIELDLOG:END-->
§04 CHANNEL ASSIGNMENTS (kept)
<!--FOOTER:START--> FILED <date> · ENTRY № 1 · THIS DOSSIER IS LIVE (re-filed <date>) <!--FOOTER:END-->
```

**The only dynamic README *markdown* is two marker sections — `FIELDLOG` and `FOOTER`.** Everything else dynamic
(seal, header, doctrine, status chits) is a regenerated **SVG file** referenced by a fixed relative path, so the
README markdown around it is stable. SVGs referenced by **relative path** only.

## 6. Correctness rules (the bugs the research verifiers caught — must hold)

1. **SVG rendering:** committed `.svg` via `<img>`/`<picture>`, relative paths. No inline SVG (stripped). No
   external badge URLs (camo caches ~31 days; relative committed assets refresh per commit).
2. **Animation:** SMIL only, `fill="freeze"`, no loop. (CSS keyframes freeze in Firefox-via-img.)
3. **Fonts:** all SVG text outlined to vector paths via `fonttools` (`svgPathPen`). Guarantees identical render
   and stable clip widths; removes font-fallback + clip-mis-registration risk.
4. **Seal hash stability:** `SEAL` hashes README **with the `FOOTER` and `FIELDLOG` marker blocks stripped**
   (those are the only clerk-rewritten markdown regions; the seal is an image file, not README text, so it
   isn't in the hash anyway). Prevents the self-referential hash → infinite-rewrite loop (reproduced in
   research; strip-fix confirmed stable).
5. **REV semantics:** `REV` counts commits **excluding** `dossier-clerk[bot]` commits, so it doesn't self-climb.
6. **Field-log decay:** no color/`style`/`font-family` (all stripped in markdown). Use `<sub>` shrink / `█`
   redaction for the "sinking into archive" effect.
7. **Idempotency / no loop / no graph spam:** commit only on real diff (`git diff --quiet || commit`),
   `[skip ci]`, bot author, generated assets excluded from REV count.
8. **Cron reality:** treat cron as "roughly daily" (best-effort, can be delayed/skipped). `workflow_dispatch`
   for manual refresh. The daily commit (when content changes) resets the 60-day schedule-disable clock; if a
   long quiet streak is a concern, a monthly keepalive can be added later (out of scope now).
9. **Honest liveness:** footer says "re-filed <date>" (cron cadence), never implies per-second live.

## 7. Testing / verification

- **Render locally** (`python -m clerk.main` against a checked-out copy) and open each SVG in **Chrome + Firefox**
  to confirm SMIL animates (Chrome+Firefox) and static pieces render; validate SVG well-formedness (xmllint).
- **Hash-stability test:** run the clerk twice in a row; assert the seal block + all rewritten sections are
  byte-identical on the second run (no churn). Automated as a small pytest.
- **Marker-replace unit tests:** `markers.replace_section` round-trips and is idempotent.
- **PAT read test:** confirm `DOSSIER_READ_TOKEN` reads a private REF's release/push date (dry-run script).
- **Prototype branch first:** view rendered README + raw `assets/*.svg` via `raw.githubusercontent.com` before
  merging to `main`. Toggle GitHub light/dark and check mobile fallback.

## 8. Phasing

1. **Scaffold + static pieces:** `clerk/` package, `dossier.yaml`, `svgtext.py` outliner, render the **static**
   seal + classification header + doctrine (static draft), restructure README. Branch: `feat/living-dossier`.
2. **Dynamics:** the Action, status reconciliation (incl. PAT), field log, and the DOCTRINE SMIL type-in.
3. **Verify** across browsers + private-repo reads; then merge to `main`.

## 9. Risks / open items

- **Outlining pipeline** is the trickiest engineering bit (bundling a monospace TTF, extracting glyph paths,
  laying out fixed-width text). Mitigation: a single well-tested `svgtext.py` used by every renderer; fall back
  to a base64-embedded subset font only if outlining proves brittle.
- **PAT scope:** must be fine-grained, read-only, limited to the specific private REF repos; stored as a repo
  secret, never committed. User creates it.
- **Camo cache lag:** edits may show the old SVG for a few minutes — acceptable for a "filed" artifact.
- **`docs/` in the profile repo:** spec + clerk code live in the repo but don't render on the profile (only the
  README does), so no visual leakage.
